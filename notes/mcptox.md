## What is MCPTox?

The first systematic benchmark for evaluating LLM-agent robustness against **Tool Poisoning attacks** in the Model Context Protocol (MCP) ecosystem.

Tool Poisoning = malicious instructions embedded in a tool's **metadata** (name, description, schema) — not in its execution output. The attack happens before the tool is ever called: the agent reads the poisoned description, treats it as legitimate instruction, and acts accordingly.

This is distinct from prior MCP-attack work which focused on **tool-output injection** (malicious data returned by a compromised tool). MCPTox attacks the surface that exists *before* execution.

## Methodology

1. Collect 45 real MCP servers and their 353 tools.
2. Design 3 attack templates (`[GAP: templates undefined]`).
3. Use few-shot prompting to instantiate templates against each tool → 1312 malicious test cases.
4. Run each test case against 20 LLM agents.
5. Measure attack success rate (ASR) and refusal rate per model.

## Key findings

- **Widespread vulnerability:** Tool Poisoning is not an edge case — every tested agent fell for it at meaningful rates.
- **o1-mini ASR = 72.8%** — the highest reported.
- **Counterintuitive scaling:** More capable models are *more* susceptible, not less. The paper's claim: stronger instruction-following ability = better at executing attacker's instructions when those instructions arrive via tool metadata.
- **Refusal is broken:** Claude-3.7-Sonnet, the most refusal-prone model, refused **< 3%** of attacks. Existing RLHF/safety alignment does not engage when the malicious instruction arrives via tool description rather than user prompt.
- **Tools-as-vector:** The attacker doesn't need to compromise execution; they just need control of a tool's *description*. This is a low-cost, high-impact attack surface.

## How this is relevant to mcp-sentinel

This benchmark is exactly the threat model mcp-sentinel targets:
- Scan tool metadata (descriptions, schemas, names) — not runtime outputs.
- Detect poisoned instructions embedded in legitimate-looking tool definitions.
- Two-stage detection (semantic similarity to known-malicious corpus → LLM judge) is the right architecture for this threat because:
  1. The attack space is open (1312 cases is a sample, not a complete enumeration) — regex/keyword detection will miss novel phrasings.
  2. Embeddings catch semantic neighbours of known attacks even when surface form differs.
  3. The LLM judge resolves the borderline cases that similarity alone gets wrong.

MCPTox gives us:
- A ready-made evaluation set (1312 known-malicious examples) once we get access.
- A baseline ASR per model that we can claim to *reduce* if mcp-sentinel sits in front of the agent.
- An external citation that justifies the scanner exists at all — "this isn't a hypothetical threat, see MCPTox: 72.8% ASR on o1-mini."

### How we'll use the dataset (when accessible)
- Hold-out evaluation: never train embeddings on MCPTox; use only as a test set to report `precision @ k`, `recall @ k`, and end-to-end blocking rate.
- Cross-corpus check: compare which MCPTox attacks our corpus *fails* on — those are the categories we under-sampled.

## Limitations of MCPTox (from what we can see)

- Only 3 attack templates. The 1312 cases are template-driven, so they likely cluster in style. A real-world attacker won't follow templates.
- Tested only LLM agents — no defenders. They establish the attack baseline; they don't evaluate scanners or guardrails.
- `[GAP]` Whether the 45 servers are still live / reproducible 6 months from now.
- `[GAP]` Whether attack templates are static or get refreshed as models patch.
